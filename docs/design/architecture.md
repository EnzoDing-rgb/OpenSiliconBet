# RISC-V 三国杀 · V2 架构设计与实施蓝图

> 本文档为 Coding Agent 的绝对执行标准。架构决策已冻结，执行层严格按此蓝图编码；与 [**implementation.md**（计划 + 测试规格合一）](./implementation.md) 配套使用。

## 产品速览（PM memo · 阶段顺序表）

观众在浏览器里经历的一条龙：**主持垫场** → **预录开场** → **三阵营圆桌**（定量收口）→ **Lex 主持引介** → **远程嘉宾视频连线**（黄仁勋）→ **转场进互动** → **自由问答**。

**状态机顺序（钉死）**：`0 → 0.5 → 1 → 2 → 2.5 → 3`（论坛交锋，非胜负辩论）。

| 顺序 | 阶段值 | 名称 | 观众看到什么 | 实现要点（摘要） |
|:---:|:---:|:---|:---|:---|
| 1 | **0** | Init | 顶栏标题、场地副标题、Lex 极简垫话 | 静态布局 + Lex 文案 |
| 2 | **0.5** | 预录开场 | Lex 长贯口音频；可一键跳过 | 长轨 →（skip）短轨「好的，那让我们直接开始。」→ 进 **1**；禁两轨叠声 |
| 3 | **1** | 论坛交锋 | 吴伟 / 陈立武 / 库克 圆桌对撞 | Baton：`@` → 隐式 → LRS；**每人 2 段、共 6 段**收口；文本可 **SSE**，**整段完稿再 TTS**，音频 **回合级播放锁** |
| 4 | **2** | 远程嘉宾入席（视频） | Lex **主持引介**（一两句书面串场）→ **视频连线** UI（纯 CSS 视窗/头像入场，无真人 MP4）→ 黄仁勋发言 | Jensen **≤200 字**；调用前 **必注** [`jensen-closing-speech.md`](../background/jensen-closing-speech.md)；小窗挂起 |
| 5 | **2.5** | Lex 转场 | 固定一句：进入观众提问环节 | **文案钉死** + TTS；**不可省略**；与黄轨 **串行**（播放锁） |
| 6 | **3** | Free QA | 主屏提问；嘉宾或 Lex 回应 | `Summary` + `Intent` 路由；黄小窗仍在席 |

**横切策略**：除 **0.5** 预录 MP3 外，发言侧为 **LLM（千问等）+ DashScope TTS**；**禁止**半句切块叠播。**2 → 2.5 → 3** 必须走同一套 **播放完成再推进** 的锁链。

**能力一句话**：多阶段状态机 + 三嘉宾自动传棒 + **整段完稿再 TTS** + Lex 口播转场 + 观众问题结构化路由。

## 1. 核心运行机制与阶段 (Phases)

产品形态从「胜负式辩论」调整为 **论坛交锋**（观点对撞、可核验、非抬杠判输赢）。叙事由后端全局状态机（`StageContext`）控制，含下列阶段（**含 2.5 转场，不可省略**）：

*   **阶段 0 (Init)**: Lex 极简开场垫话 + 顶栏标题（RISC-V 三国杀）与场地副标题（公众科学日分会场）。
*   **阶段 0.5 (Pre-recorded Opening)**: 播放预先合成的 Lex 贯口（`lex-opening-long.mp3`，由 `scripts/pregen_lex_opening.py` 生成，音色默认 `VOICE_ID_LEX`）。支持前端一键“跳过”，跳过时立刻切断长轨，播放 `lex-opening-short.mp3`（“好的，那让我们直接开始。”），随后无缝切入阶段 1。
*   **阶段 1 (Forum — 论坛交锋)**: 吴伟 (RISC-V)、陈立武 (x86)、库克 (ARM) 三人交锋；传棒仍用 Baton（见第 3 节）。**交锋截止（钉死）**：**每位嘉宾各完成 2 次发言**（各 2「轮话」），即 **共 6 条** 嘉宾发言 turn（仅计 `wuwei|liptan|cook`，不含 Lex）。第 6 条发言结束并播放完毕后，**立即**进入阶段 2，不再追加第七段三人交锋。
*   **阶段 2（远程嘉宾 · 视频连线）**：Lex **主持引介**（书面串场，宣布请远程嘉宾上线；语气对齐高端论坛，非综艺「响铃」梗）→ **视频连线** 界面：**纯 CSS**（视窗展开、头像入场等，无真人 MP4）→ 黄仁勋 (Jensen) 独白（≤200 字）→ 小窗保留。调用 Jensen 前 **必须**读入并注入 [`../background/jensen-closing-speech.md`](../background/jensen-closing-speech.md)（弹药：锚点 + 金句池 + 独白骨架），与 Jensen SKILL、`deep-research.md` 同屏进 prompt。独白音频 **`onended`**（或等价播放完成信号）后，**必须先**进入 **2.5**，再进阶段 3。
*   **阶段 2.5 (Lex 转场口播 — 必选)**: 黄仁勋发言结束后，**Lex 口播一句**固定转场，**文案钉死**：「好，观众朋友们，我们现在进入观众提问环节。」实现上可为 **固定字符串 + TTS**（与全场过渡期同一 `VOICE_ID_MEARSHEIMER` 亦可），或后续改为 Lex 独立声线；**不得**依赖用户即兴或跳过（除非 debug 旗）。本句播放完毕后再进入阶段 3。
*   **阶段 3 (Free QA)**: 主屏开放 User Chat。黄仁勋小窗保留，与另外三人共同在席，接受用户提问。

## 2. TTS 与流式并发控制（严禁重叠）

**痛点解决**：彻底废弃“文本切块 -> 实时 TTS”的激进方案，以解决上下句音轨重叠、抢播的问题。

**强制执行标准（Turn-level Blocking）**：
1. **文本流式 (SSE)**：LLM 推理时，文本通过 SSE 正常流式推送到前端，保证打字机视觉体验。
2. **音频后置合成**：在当前 Turn 的 LLM 文本**完全生成完毕**（拿到 `finish_reason: stop` 和完整字符串）后，将**整段文本**一次性发给 DashScope TTS 进行合成。
3. **播放锁 (Playback Lock)**：前端拿到完整音频流后开始播放。必须等待当前音频 **`onended` 事件触发**，前端才允许向后端请求下一个 Speaker 的发言（或者后端严格等待前端的 `audio_finished` 信号后再推进状态机）。
*   **妥协点**：牺牲了单句音频的首字节延迟（TTFB），换取绝对稳定的顺序播放和零重叠。

## 3. “抢话与传棒”逻辑 (Baton Logic)

**痛点解决**：废弃“无人接话就交回给主持人”的生硬逻辑，模拟真实圆桌的“火药味”和自动抢话。

**强制执行标准（三级抢话机制）**：
当当前嘉宾（如 A）发言结束时，按以下优先级决定下一个发言人：

*   **优先级 1：显式点名 (Explicit `@`)**
    *   正则扫描 A 发言的最后一行或全文，若存在明确的 `@吴伟`、`@库克`、`@x86` 等（需做别名归一化），则直接将棒交给他。
*   **优先级 2：隐式挑衅 (Implicit Provocation - 核心机制)**
    *   若无显式 `@`，后端调用一个极轻量的 `NextSpeakerSelector` LLM（输入：A 的最新发言内容；Prompt：“根据这段话的内容，剩下的两位嘉宾谁最有可能被激怒、被内涵，或者最需要为自己的阵营辩护？只输出名字”）。
    *   **效果**：即使 RISC-V 没点名，只要话里疯狂拉踩 x86，x86 嘉宾就会自动“抢话”。
*   **优先级 3：最久未发言兜底 (Least Recently Spoken, LRS)**
    *   如果 LLM 选择失败或认为无人被挑衅，系统自动将棒交给**在场嘉宾中，距离上次发言间隔最久**的那个人，确保每个人都有足够的戏份。
*   *(注：Lex 作为 Host，在阶段 1 不参与抢话，只在阶段流转时介入)*。

## 4. 自由问答管线 (Free QA Pipeline)

在阶段 3，用户的每一次输入必须经过 `free_qa.py` 管线处理：

1. **SummaryLLM (上下文浓缩)**：
    *   提取最近 3 轮的 verbatim 原文。
    *   生成增量摘要（≤300字，`temperature=0`），防止长程对话导致 Token 爆炸。
2. **IntentTargetLLM (意图与分发)**：
    *   分析用户输入，输出 JSON：`{ "intent": "ASK"|"PROVOKE"|"CHITCHAT", "target": "wuwei"|"liptan"|"cook"|"jensen"|"lex", "confidence": 0.9, "rephrased_question": "..." }`
3. **执行分发**：
    *   若 `target` 明确，将 `rephrased_question` 发给对应嘉宾的 Prompt 生成回答。
    *   若识别失败、遇到越界/攻击类输入，统一路由给 Lex 进行软回应（“这个问题超出了我们今天的讨论范围...”）。

## 5. 音色分配 (Voice IDs)

五辩手（`lex` / `wuwei` / `liptan` / `cook` / `jensen`）各自一条 **`VOICE_ID_*`**，写在项目根 `.env`。未单独配置的角色会退回 `VOICE_ID_DEFAULT` → `VOICE_ID_MEARSHEIMER` → `VOICE_ID_JERVIS`（见 `backend/app.py`）。

批量复刻（北京、`qwen-voice-enrollment` + `target_model` = `TTS_MODEL` 同款 realtime）：把 `Lex.m4a`、`Wuwei.m4a`、`Lip-Bu.m4a`、`Tim_Cook.m4a`、`Jensen_Huang.m4a` 放在 `assets/`，运行 `python3 scripts/enroll_voices.py`（读 `DASHSCOPE_API_KEY`，自动回写上述五键）。控制台手工上传亦同，须保证 **`target_model` 与 `TTS_MODEL` 完全一致**。

历史双人素材：`VOICE_ID_JERVIS` / `VOICE_ID_MEARSHEIMER` 仍可保留作回退或预录合成。

## 6. 全局约束 (Global Constraints)

*   **文本长度控制**：所有 LLM 的 System Prompt 首部必须注入 `GLOBAL_SPEAKER_CONSTRAINT`：“你的输出正文必须控制在 200 字以内（JSON 字段或单独一行的 @ 标记不计入字数）。”
*   **前端审美**：5+1 布局。阶段 1 为三人等宽排布；阶段 2 触发 CSS 弹窗；阶段 3 为四人排布 + 问答区。UI 需保持极简科技风。

## 7. 自动化测试（概要）

*   **详规**：[implementation.md](./implementation.md) 第二部分（按阶段列单测 / mock LLM / 播放锁 / 省钱策略）。
*   **原则**：默认 **pytest + Vitest 离线**；全链路 LLM/TTS 仅在 opt-in 环境变量下 **极少调用**；CI 不打真实大模型。

## 8. 人眼验收只做这三四点

交付后人工只看这些是否「像 demo」即可，其余交给自动化。

1. **0.5 预录**：长开场能播；点跳过 → 立刻停长、播短句、再进入对谈，**无两轨叠声**。
2. **阶段 1 论坛交锋**：节奏像圆桌对撞；**数得清**三人各说了两段（共六段）后自然进入 **视频连线** 环节；**无上一句没播完下一句就抢播**；`@` 传棒符合预期。
3. **阶段 2 + 2.5**：有 **视频连线 / 远程入席** 级 UI 动效 + 黄仁勋小段话；**黄说完之后** Lex **必定**接「好，观众朋友们，我们现在进入观众提问环节。」再出现提问 UI（不可少句、不可与黄轨叠播）。
4. **阶段 3 问答**：主屏发几条问题，能稳定落到某一嘉宾或 Lex 软兜底，且界面不崩、不串台。

**已钉死（与 §1 一致）**：阶段 1 → 2 的收口 = **6 条嘉宾发言**（每人 2 条）；常量建议 `FORUM_GUEST_TURNS_TOTAL=6`（或等价计数器），写入 `.env.example` 时仅作说明，**默认勿改**除非主办方明确要求加时。

## 9. 三家股价联动（市场快照 · 设计意图）

**要做什么（产品一句话）**：在观众界面暴露一条**只读、带时间戳**的「三家代表性上市公司」行情快照（或轻量 ticker strip），让论坛交锋里的**阵营叙事**与**资本市场当下情绪**对齐——嘉宾可以像「看看今天盘面」那样把波动纳入论证，观众一眼看到「钱在往哪边叙事投票」。

**为什么是「三家」**：与「RISC-V / x86 / ARM」三条故事线挂钩，各选**一只最具符号意义的流动性标的**作为代理（具体 ticker 表可配置，不要求写死在架构里）。示例心智模型（实现时可改表，但语义不变）：

| 叙事轴 | 典型代理 | 观众心智 |
|--------|----------|----------|
| x86 / 传统 PC–服务器 CPU 叙事 | 如 **INTC**（或主办认可的 x86 生态代理） | 「老帝国」估值与反转预期 |
| ARM / 低功耗授权与终端–Apple Silicon | 如 **ARM** 或 **AAPL**（二选一或并列由配置决定） | 「授权 + 终端溢价」是否在定价里 |
| AI / 加速器与「算力军备」外溢 | 如 **NVDA**（或 QCOM 等，由 deep-research 口径约束） | 「Agent 时代谁吃增量」 |

**不做什么（边界）**：

- **不做**交易、下单、投资建议；UI 文案固定为「公开行情快照 · 仅供讨论语境」类免责声明。
- **不做**高频实时逐笔刷新把 LLM 上下文刷爆；默认 **会话开局抓一次快照**（或低频定时，如 5–15 分钟），注入到可选 `market_context` 块；论坛进行中除非显式「刷新行情」按钮，否则不自动重拉。
- **变动极快的数字**须遵守 [`../background/deep-research.md`](../background/deep-research.md) 事实口径总注：模型输出里引用行情时带 **截至时间**，避免把单日波动写成结构性结论。

**与叙事的关系**：行情块是**佐料**——`free_qa` / 论坛 prompt 可选择拼接「截至 T 的涨跌幅、市值量级」，让角色扮演更「在场」，但不改变 baton、阶段机、TTS 播放锁等主干。

**验收（设计层）**：页面上能解释清楚「这三家为什么出现在同一行」；无 key 泄露（密钥只走后端）；断网/限流时降级为「上次快照时间 + 不可用」而非空白崩溃。

## 10. 观众 Chat 与观点持久化（LLM 策展 → 落盘文档）

**要做什么（产品一句话）**：阶段 3 观众 **chat 照常 ephemeral**（会话内列表）；另提供**显式按钮**（如「把本场金句写进笔记」），把**截至当前**的嘉宾发言（含论坛交锋与 Free QA 中已产生的 turn）+ 可选观众问题摘要，打包送给 **Curator LLM**，由模型**挑选、去重、改写为可引用笔记**，再**写入仓库内指定 Markdown**（例如 `docs/background/session-notes/` 或主办约定路径），实现「犀利观点落盘」而非全盘聊天记录 dump。

**用户动线**：

1. 辩论 / 问答进行中：chat 区行为与现有一致（不要求每条观众消息都持久化）。
2. 主持人或操作员在适当时机点击 **持久化** 按钮（可多次点击 = 多次 append 或 merge，由实现约定）。
3. 后端组装 payload：`recent_guest_turns`（带 `speaker`、时间或 turn index）、`optional_audience_digest`、`session_id`、已有笔记文件头（防重复段落）。
4. **Curator LLM** 输出结构化结果（建议 JSON）：`items[]` 每项含 `title`、`quote_or_paraphrase`、`why_valuable`、`source_turn_refs`；过滤辱骂、纯灌水、与事实冲突的断言可标 `low_confidence` 或丢弃（规则写进 system prompt）。
5. **落盘**：将 JSON 渲染为固定模板的 Markdown 区块（带 UTC 时间、run_id）**追加**到目标文件；Git 是否提交由人工作流决定，应用本身只负责写文件或写 API 再由 CI 提交（实现阶段再选）。

**明确延后（本设计不钉死）**：Dev server `Ctrl+R` 刷新后是否自动拉取最新落盘内容、是否 SSE 通知前端「笔记已更新」——**留待后续**；当前只要求「写文件成功可验证」与「路径可配置」。

**安全与成本**：

- 按钮触发 = **显式用户同意**，避免每条消息都打 LLM。
- 对落盘路径做 **allowlist**（仅项目内 `docs/...` 子树），禁止任意路径写。
- Token：策展上下文应截断（如最近 N 字 + 摘要），与 `free_qa` 摘要策略一致。

**验收（设计层）**：一次点击后目标 md 出现新增一节；重复点击不产生无限重复段落（或实现 idempotency key）；Curator 失败时 UI 报错且**不写半篇**。

**与 TTS / 实时音频**：正交；落盘不依赖音频完成事件，仅依赖已提交的文本 turn 日志。

## 11. 共享事实底盘（`deep-research.md`）· Token 策略（未来优化）

**现状（实现已落地）**：`debate_runner._build_system_prompt` 将 **SKILL + `deep-research.md` 全文** 拼进 system；**论坛每一轮、Jensen 串场、`chat_with_debater` 每次调用** 各付一遍完整 input（除非供应商对**相同前缀**自动做 **prompt cache**，代码侧**未假设**命中）。Jensen 串场另有 **system 字符串长度上限**（尾部截断），过长时可能削掉档案尾部。

**优化目标**：在 **不明显丢可引用事实** 的前提下，降低 **input token × 调用次数**；避免「只首轮注入 → 后几轮模型瞎编数字」。

**推荐路线（可组合，按性价比排序）**

1. **稳定前缀 + 缓存友好拼接（低成本）**  
   若上游 API 支持 **静态 system 前缀缓存**（OpenAI / 部分兼容网关等）：把 **全文不变块** 放在 **每条请求 system 的最前部且字节级稳定**（例如先 `deep-research.md` 全文，再拼 **各嘉宾 SKILL**）。同一 run 内 DR 全文不变 → 缓存命中时 **后续轮次主要只为 SKILL 差分付费**（以供应商计费规则为准）。**注意**：当前实现是「SKILL 在前、DR 在后」，若要试缓存需 **改拼接顺序** 并做 A/B 测引用质量。

2. **双层档案（推荐主方案）**  
   - **L0 铁律卷**（目标 **~1–2k tokens**）：从 §6 关键数字、§7 语录、专题摘要各抽 **短句级** 条目，带 **§ 锚**；禁止项（勿混用 65% / 90.6% 等）写进同一小页。构建方式：人手维护 `deep-research-compact.md` **或** CI 从正文生成（diff 可读）。  
   - **L1 全文**：保留现有 `deep-research.md`。  
   **论坛策略**：例如 **第 1 轮（或每位嘉宾首次开口）注入 L0+L1**；**同阶段后续轮仅 L0**，并在 system 加硬句：「具体展开与反方论据以 L0 中 § 锚为准；**勿发明 L0 未列数字**；若需引用长论证可口语说『档案里有更细的』而不展开。」**阶段切换**（进 Jensen、进 Free QA）可 **再叠一次 L1** 或 **按阶段注入 L1 头尾**（如仅「事实口径总注」+ 专题短稿），防跨阶段遗忘。  
   **收益**：6 轮论坛里 **约 5 轮** 省掉大半 L1；**可引用性**靠 L0 的 § 锚 + 角色 SKILL 里对 § 的写作约束维持。

3. **按意图检索（Free QA 优先）**  
   已有 `SummaryLLM` / `IntentTargetLLM`。扩展：对 **rephrased_question** 做 **轻量关键词或 embedding**，从 `deep-research.md` 分块（按 `## §` 切）取 **Top-K 段** 拼进回答嘉宾的 system/user，而非每次全文。论坛交锋若要做检索，需额外 **话题状态**（当前争点）避免漏块——工程量比 2 大。

4. **离线压缩管线（与 2 配套）**  
   用强模型对 L1 做 **保留所有数字与日期** 的压缩稿，人工 diff 后晋升 L0；或双文件并行维护（L0 手 curated，L1 长文）。

**不推荐单独使用**：「整盘辩论只有首轮给全文、其余轮零档案」——除非 L0 极厚或模型极强，否则 **SPEC/股价级幻觉** 风险高。

**验收（将来实现时）**：同议题 A/B，盲评 **数字错误率**、**§ 引用一致性**、**input token 曲线**；Jensen 串场 **不得**因截断丢掉 §6 头部数字区（若实现 L0，应保证 L0 在截断窗口内）。
