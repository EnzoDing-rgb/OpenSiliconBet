# 实时语音合成架构方案（逐轮发言 · 自动播放）

目标：每轮由 **一名** 发言人产出整段正文 → 后端用阿里云「千问实时TTS」按该轮的 `voice_id` 合成 → 经 WebSocket 把音频块推到浏览器自动播放。早期文档以「固定顺序、双人、各三轮」举例；**RISC-V 三国杀** 改为 **v2 Director + 论坛交锋 + 三级 baton**、多发言人与（可选）多套 `voice_id` 时，**管道不变**（仍是一轮一人、一块文本 → TextChunker → TTS → 转发）。**产品级 TTS 策略**（整段完稿再合成、播放锁防叠播）以 [`architecture.md`](architecture.md) 为准。

**与其他文档**：架构宪章见 [`architecture.md`](architecture.md)；**计划 + 测试**见 [`implementation.md`](implementation.md)。事实与人设见 [`../background/deep-research.md`](../background/deep-research.md) 与 [`../characters/`](../characters/) 下各 `*-perspective-SKILL.md`。Lex **0.5 预录**（长/短 MP3）不走本节实时 WebSocket，见 [`../background/lex-opening-script.md`](../background/lex-opening-script.md)。LLM 侧 token/摘要策略（与 TTS 正交）见归档 [`../_archieved_mds/prompt-token-optimization.md`](../_archieved_mds/prompt-token-optimization.md)。

## 前置：你在阿里云平台要做什么（一次性准备）

这是你必须先做完的，不然代码跑不起来：

### 1. 开通服务 + 拿 API Key
- 开通：阿里云「百炼/Model Studio」里 **语音合成** + **声音复刻/声音设计** 两个服务
- 在控制台生成一个 **API Key**，记下来 → 放后端环境变量里，**绝对不能放到前端代码里**（会泄露，别人刷你钱）

### 2. 复刻辩手音色（你有音频）
- 在「语音模型 → 声音复刻/声音设计」页面，为每位需要克隆的发言人各上传一段音频（下文以**双人示例**「杰维斯 / 米尔斯海默」命名；五辩手时同理准备五份 enrollment，拿到五个 `voice_id`）。
- 注意：
  1. 音频满足要求（千问声音复刻 `qwen-voice-enrollment` 的官方口径）：
     - 支持格式：WAV(16bit)/MP3/M4A
     - 时长：推荐 10~20 秒，最长不超过 60 秒
     - 文件大小：< 10MB
     - 采样率：≥ 24 kHz
     - 声道：单声道
     - 内容：至少 3 秒连续清晰朗读（无背景音），允许短暂停顿（≤2秒），避免背景音乐/噪音/他人声
  2. **必须选对「目标合成模型（target_model）」**：你在复刻时选择的 `target_model`，后续实时合成时的 `model` 必须完全一致，否则一定失败。
     - 推荐实时模型：`qwen3-tts-vc-realtime-2026-01-15`
  3. 生成完成后，你会拿到每个辩手一个音色标识（`voice` / 常被叫作 voice_id）。双人示例：
     - `voice_jervis`
     - `voice_mearsheimer`
     - 记下来，写到后端配置里（多人时同理扩展为多条 env 或配置表）。

> **RISC-V 三国杀 Lex 0.5**：预录长/短轨可用现有 `VOICE_ID_MEARSHEIMER` 合成，见 [`../background/lex-opening-script.md`](../background/lex-opening-script.md)；与本节的「论坛/实时 TTS」是两条线。

### 3. 确认实时合成模型名称
就用你复刻时选的目标模型名，比如：
- `qwen3-tts-vc-realtime-2026-01-15`

## 为什么要这么设计：后端转一道手，不直连浏览器

有两个最关键原因，新手很容易踩坑：

1. **API Key 绝对不能放浏览器**：放前端一打开就泄露，别人能拿着你的 key 随便调用刷钱
2. **后端适合做中间层**：统一限流、重试、日志、保存回放，浏览器只需要专心播放音频就行

## 完整走一遍流程（自动播放，不需要用户点）

当「下一轮由谁说」已由后端决定（固定表、论坛交锋 Director、或 baton 解析结果）时，流程非常简单：

1. 用户在浏览器点「开始辩论」（浏览器要求必须用户点一下才能解禁音频播放，安全策略要求，没办法）
2. 后端启动辩论 → 自动一轮一轮走：
   - 选当前轮发言人 → 拿到对应 `voice_id`
   - 调用 LLM 得到 **完整一整块发言文本**（你现在本来就是这么做的）
   - 把整块文本切分成小段子（这步就是 `TextChunker`，下面讲为什么切、怎么切）
   - 连阿里云 TTS WebSocket → 把切好的小段子逐块发上去
   - 阿里云返回音频块 → 后端转发给浏览器 → 浏览器马上播放
   - 当前轮全部走完 → 通知浏览器「这轮完了」 → 自动开始下一轮
3. 全部辩论轮次都走完（旧双人稿常写「三轮」；多辩手时轮数由配置决定）→ 如果需要，裁判评分也可以合成语音（可选）
4. 全程：自然就是「同一时刻只有一个发言人在说话」，不用额外加锁控制，非常简单

## 核心模块：TextChunker（为什么切？怎么切？）

### 为什么要切？
- 你现在 LLM 是一次性返回一整块文本，如果直接把整段发给 TTS 再一次性拿音频，用户要等很久才能听到声音
- 切分成小块，一块出来就一块播放，用户很快就能听到声音，体验好很多
- 如果不切，TTS 也要做切分，而且整段太大更容易超时

### 最简单可解释的切分规则（就是你说你喜欢的这个，直接用）
1. 开一块缓冲区，逐字符放进去
2. **什么时候发出去（flush）**，满足任意一个条件就发：
   - 缓冲区字符数累计到 `min_chars` → 默认 40~80 个中文字符 → 可以发了
   - 遇到句末标点：`。！？；\n` → 马上发，不等了
   - 缓冲区字符数超过 `max_chars` → 默认 120~200 → 必须发，避免块太大导致延迟
3. 发出去之后清空缓冲区，继续攒下一块

这样做既能保证“用户不用等太久就能听到声音”，又能保证语音韵律自然（断句对了，语音读起来才像人）。

## 后端内部模块（每个模块做什么）

### 1. TextChunker（就是上面讲的切分）
职责：把整段文本切成适合 TTS 播放的小块，平衡延迟和韵律。

### 2. SpeakerManager（天然适配「一轮一人」）
- 无论发言顺序来自固定表还是动态 baton，**同一时刻只有一个活跃 TTS session**，逻辑不变
- 只需要加一点保险：每一轮开始前，确保上一轮资源已经释放（关闭 WebSocket 这些）

### 3. TtsSession（状态机，不会乱）
每个发言对应一个 session，状态不会乱：
```
IDLE（空闲） → CONNECTING（连阿里云） → STREAMING（发文本收音频） → DRAINING（发完文本等最后音频） → CLOSED（关完了）
                         ↘ ERROR（错了） ↗
```
每一步该做什么清清楚楚，出了错也能干净收拾资源。

### 4. Forwarder（转发音频）
阿里云返回音频块 → 后端直接二进制转发给浏览器 WebSocket → 浏览器直接播，不用改。

## 接口怎么设计（浏览器 ↔ 后端）

### WebSocket 唯一通道（实时最佳选择）
浏览器连：
- `WS /ws/debate-audio?run_id=<run_id>`

浏览器 → 后端 只需要一种消息：
- `{"type":"start"}` → 开始自动逐轮播放，没别的。

后端 → 浏览器 几种消息：
1. 每一轮开始先发 `meta`：告诉浏览器格式、谁说话、第几轮（`speaker` 为后端枚举字符串，下例沿用旧双人名；五辩手时换成 `wuwei` 等即可）：
   ```json
   {"type":"meta","format":"PCM_24000HZ_MONO_16BIT","speaker":"jervis","round":1}
   ```
   浏览器需要这个知道「怎么配置音频播放」「高亮谁」
2. 之后就是二进制音频块 → 浏览器直接喂给播放器
3. 当前轮结束发：
   ```json
   {"type":"round_done","speaker":"jervis","round":1}
   ```
4. 错了发：
   ```json
   {"type":"error","message":"具体错误信息"}
   ```
5. 全部轮次走完发（消息名仍可为 `all_done`，与具体轮数无关）：
   ```json
   {"type":"all_done"}
   ```

## 浏览器这边要做什么（人话要点）

1. **播放器**
   - PCM 音频需要一个「环形缓冲区（ring buffer）」存数据，避免卡顿，这个是音频播放常识
   - 两种实现难度：
     1. `ScriptProcessorNode`：简单，虽然标记 deprecated，但用来快速验证完全没问题
     2. `AudioWorklet`：推荐上生产用，延迟更低更稳定
   - 你不用一开始就完美，先跑通再优化
2. **用户交互**
   - 只需要用户点一下「开始辩论」（浏览器安全策略要求）
   - 不需要任何暂停/播放/逐轮控制按钮，自动走就行
   - 用户不想听直接静音系统音量就行（你说你 mac 自己有静音键，所以不用加按钮）
3. **文字高亮**
   - 收到 `meta` 就把当前轮/当前发言人高亮，用户一眼看到哪了
   - 文字滚动跟上音频进度就行，不用精确逐字对齐，音频出来滚一下就行

## 最容易踩的坑（提前告诉你，别踩）

1. **voice_id 模型一定要匹配**：你复刻的时候选的 target_model 必须和合成用的模型一模一样，错了一定失败
2. **不切分直接喂整段**：用户要等很久才能听到声音，体验很差 → 一定要切分
3. **不释放资源**：每一轮完了一定要关闭阿里云 WebSocket，不然连接泄漏越来越多
4. **API Key 放前端**：泄露了你就被扣钱，一定要放后端

## 最小落地步骤（一步步来，每一步都能验证）

推荐你按这个顺序做，错了马上能定位：

1. 在阿里云控制台复刻完所需辩手音色（可先双人 smoke），先自己控制台试听对不对，确保没问题
2. 后端写一个最小测试：固定一段文字 → 走 TTS → 浏览器播放，不接辩论LLM，先把通路跑通
3. 接进去辩论自动流程：一轮一轮调用 TextChunker + TTS → 转发
4. 前端加高亮 + 播放，搞定
5. 最后看要不要加裁判评分语音（可选）

这样每一步都能验证，出了问题也好修。
